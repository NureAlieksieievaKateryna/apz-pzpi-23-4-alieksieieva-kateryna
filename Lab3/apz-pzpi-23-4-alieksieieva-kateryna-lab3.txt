2	ЗАВДАННЯ

У межах даної лабораторної роботи було розроблено front-end частину програмної системи «Subscription Platform», яка призначена для управління підписками між клієнтами та бізнесами. Веб-застосунок реалізовано таким чином, щоб забезпечити окремі інтерфейси для різних категорій користувачів відповідно до їхніх ролей у системі. Для звичайних користувачів реалізовано інтерфейс, який надає можливість переглядати доступні бізнеси та тарифні плани, оформлювати підписки на послуги, переглядати інформацію про власні активні підписки та контролювати їхній поточний статус. Також користувач може отримувати інформацію про термін дії підписки та доступні можливості обраного тарифного плану. Для бізнес-користувачів створено окремий інтерфейс керування підписками. Бізнес має можливість створювати нові тарифні плани, змінювати їх параметри, активувати або деактивувати підписки, а також переглядати статистику використання сервісів та інформаційну панель із загальними показниками діяльності. Для адміністраторів системи реалізовано спеціальний інтерфейс адміністрування. Він дозволяє виконувати модерацію нових бізнес-акаунтів, підтверджувати або відхиляти заявки на реєстрацію бізнесів, переглядати статистичну інформацію про роботу системи та здійснювати контроль за функціонуванням платформи. Крім того, для адміністратора реалізовано модуль масштабування та тестування навантаження, який дозволяє генерувати конфігурації Docker Compose, Kubernetes та K6 для аналізу продуктивності системи. Під час розробки застосунку було передбачено підтримку інтернаціоналізації та локалізації. Інтерфейс системи підтримує українську та англійську мови, а також забезпечує коректне відображення дати та часу відповідно до регіональних налаштувань користувача. Додатково враховано особливості сортування текстових даних та форматування інформації залежно від обраної локалі. Front-end частина взаємодіє із серверною частиною системи через REST API, реалізований на платформі Spring Boot. Обмін даними між клієнтською та серверною частинами відбувається за допомогою HTTP-запитів у форматі JSON. Для забезпечення безпеки доступу використовується механізм автентифікації та авторизації на основі JWT-токенів. Для опису архітектури системи та її функціональних можливостей були побудовані UML-діаграми, серед яких UML-діаграма прецедентів, UML-діаграма компонентів, UML-діаграма класів та UML-діаграма послідовностей процесу оформлення підписки. Вихідний код програмної системи розміщено у репозиторії GitHub для подальшого супроводу та демонстрації результатів роботи.

3	ОПИС ВИКОНАНОЇ РОБОТИ

Під час виконання лабораторної роботи було вирішено завдання проєктування та реалізації веб-клієнтського застосунку для платформи керування підписками Subscription Platform. Розробка була зосереджена на створенні сучасного веб-інтерфейсу для взаємодії клієнтів, бізнес-користувачів та адміністраторів системи. Для реалізації клієнтської частини використовувалися технології HTML5, CSS3 та JavaScript із застосуванням REST API для взаємодії із серверною частиною системи.
Розроблений веб-застосунок забезпечує роботу трьох основних категорій користувачів. Для клієнтів реалізовано можливість перегляду доступних бізнесів та тарифних планів, оформлення підписок, перегляду активних підписок і контролю їх поточного статусу. Для бізнес-користувачів створено інтерфейс управління підписками, який дозволяє створювати нові тарифні плани, редагувати їх параметри, активувати та деактивувати шаблони підписок, а також переглядати інформаційну панель бізнесу зі статистичними показниками.
Для системних адміністраторів реалізовано окремий інтерфейс адміністрування, який забезпечує модерацію бізнес-акаунтів, підтвердження або відхилення заявок на реєстрацію бізнесів, перегляд статистичної інформації про діяльність платформи та управління системою. Додатково було реалізовано модуль моніторингу навантаження та масштабування інфраструктури, який дозволяє генерувати конфігураційні файли Docker Compose, Kubernetes та сценарії навантажувального тестування K6 для оцінки продуктивності системи.
Особливу увагу було приділено підтримці інтернаціоналізації та локалізації. Веб-застосунок підтримує українську та англійську мови інтерфейсу, забезпечує коректне форматування дат і часу відповідно до регіональних налаштувань користувача та підтримує відображення текстової інформації залежно від обраної локалі.
У процесі розробки було реалізовано інтеграцію із серверною частиною системи, побудованою на платформі Spring Boot. Взаємодія між front-end та back-end частинами здійснюється за допомогою HTTP-запитів у форматі JSON. Для забезпечення безпеки доступу використовується механізм автентифікації та авторизації на основі JWT-токенів, який забезпечує розмежування прав доступу між ролями CLIENT, BUSINESS та ADMIN.
У результаті виконання лабораторної роботи було створено повнофункціональний веб-клієнтський застосунок, який забезпечує ефективну взаємодію всіх категорій користувачів із платформою підписок та надає засоби адміністрування й моніторингу роботи системи.
3.1	Архітектура веб-застосунку

Архітектура застосунку побудована за компонентним принципом із розмежуванням відповідальності та складається з чотирьох логічних шарів: Presentation Layer, State Layer, Service Layer та Security Layer.
Шар презентації реалізований за допомогою HTML5, CSS3 та JavaScript і містить сторінки Login, Register, User Dashboard, Business Dashboard, Subscription Templates, Active Subscriptions та Admin Panel. Навігація між сторінками здійснюється через клієнтські маршрути та окремі HTML-інтерфейси для кожної ролі користувача.
Шар управління станом відповідає за збереження JWT-токена, інформації про поточного користувача та налаштувань інтерфейсу. Дані авторизації зберігаються у localStorage та використовуються для автоматичного додавання токена до HTTP-запитів.
Шар сервісів реалізує взаємодію із REST API Spring Boot застосунку. Для роботи використовуються спеціалізовані модулі UserService, BusinessService, SubscriptionService та AdminService, які забезпечують отримання та відправку даних між клієнтською та серверною частинами системи.
Шар безпеки побудований на JWT-аутентифікації та забезпечує розмежування доступу між ролями CLIENT, BUSINESS та ADMIN.
3.2	Реалізація клієнтського інтерфейсу для ролі Client

Роль Client є основною роллю системи та забезпечує роботу з підписками.
Після проходження реєстрації та авторизації користувач отримує доступ до списку доступних тарифних планів. Дані завантажуються через GET-запит до ендпоінта /subscriptions/user/templates.
Користувач може переглядати опис тарифного плану, його вартість, тривалість дії та перелік доступних функцій. Для оформлення підписки використовується форма створення підписки, яка надсилає POST-запит до /subscriptions/user/create-subscription.
Сторінка активних підписок дозволяє переглядати всі активні підписки користувача через ендпоінт /subscriptions/user/{userId}/active. Для кожної підписки відображається дата початку, дата завершення, поточний статус та залишок часу до закінчення дії.
Також реалізовано сторінку перевірки статусу підписки через ендпоінт /subscriptions/user/status/{userSubscriptionId}.
3.3	Реалізація бізнес-панелі для ролі Business
Бізнес-панель Бізнес-панель доступна для користувачів із роллю BUSINESS та призначена для управління тарифними планами.
Основною сторінкою є Business Dashboard, яка відображає інформацію про бізнес, контактні дані, статус верифікації та список створених шаблонів підписок.
Бізнес-користувач може створювати нові шаблони підписок через POST-запит /subscriptions/business/template/create, редагувати параметри підписок через PUT-запит /subscriptions/business/{templateId}/update, а також активувати або деактивувати тарифні плани через PATCH-запити /subscriptions/business/{templateId}/activate та /subscriptions/business/{templateId}/deactivate.
Для кожного тарифного плану відображається його назва, опис, вартість, тривалість та поточний статус.
3.4	Реалізація адміністративної панелі для ролі Admin

Адміністративна панель доступна за маршрутом /admin та забезпечує централізоване управління платформою.
Вкладка модерації бізнесів дозволяє отримувати список неверифікованих бізнесів через GET-запит /admin/checkAllNotVerifiedBusiness.
Для кожного бізнесу адміністратор може підтвердити або відхилити заявку через POST-запит /admin/moderate-business.
Також реалізовано вкладку масштабування системи, яка дозволяє задавати кількість backend-реплік, налаштування Kubernetes Horizontal Pod Autoscaler та параметри навантажувального тестування. На основі введених параметрів генеруються конфігураційні файли Docker Compose, Kubernetes та сценарії K6.
3.5	Підсистема інтернаціоналізації та тематизації

Підтримка інтернаціоналізації реалізована для української та англійської мов.
Усі елементи інтерфейсу, включаючи назви сторінок, кнопки, повідомлення та статуси підписок, відображаються відповідно до обраної мови користувача.
Дата та час форматуються відповідно до регіональних налаштувань користувача. Обрана мова зберігається у localStorage та автоматично відновлюється під час повторного відкриття застосунку.

Система забезпечує централізоване керування доступними тарифами та їх параметрами через зручний веб-інтерфейс.
