2.	ЗАВДАННЯ 
На цій лабораторній роботі потрібно розробити мобільний програмний застосунок для будь-якої поширеної на сьогодні мобільної платформи. Мобільний застосунок повинен бути частиною системи і повинен взаємодіяти з нею. Для реалізації програмного застосунку можна використовувати технології для реалізації програмного забезпечення для мобільних платформ, вказані лектором, а саме Kotlin для Android та Swift для iOS. Звіт повинен містити опис прийнятих інженерних рішень. У ньому повинна бути показана будова мобільного застосунку та його взаємодія з іншими частинами системи. 
 
 
 
 
 
 
 
 
 






3.	ОПИС ВИКОНАНОЇ РОБОТИ 
У процесі виконання лабораторної роботи було розроблено мобільний застосунок для системи управління підписками. Основну увагу приділено створенню інтерфейсу користувача, реалізації переходів між екранами та підключенню базової логіки взаємодії користувача із системою.
Було створено екрани авторизації та реєстрації, а також окремі головні екрани для звичайного користувача та бізнес-користувача. Для користувача реалізовано перегляд активної підписки, інформації про тарифний план, використання лімітів, історії платежів і доступ до пошуку бізнесів. Для бізнес-користувача розроблено панель керування, яка містить інформацію про бізнес, поточний план, статистику активних підписок, клієнтів, доходів і платежів.
Також було реалізовано кнопку виходу з облікового запису, яка очищає збережені дані авторизації та повертає користувача на екран входу. Для забезпечення зручності використання було застосовано сучасний дизайн інтерфейсу з картками, кнопками швидких дій, кольоровими блоками та логічною структурою розміщення елементів.
У результаті виконання роботи було створено прототип мобільного застосунку, який демонструє основні сценарії роботи системи управління підписками та може бути використаний як основа для подальшої інтеграції з серверною частиною.
3.1 Опис взаємодії компонентів web-застосунку одне з одним та з іншими елементами системи
Система FlexiSub складається з кількох основних компонентів: мобільного застосунку, backend-сервера, бази даних PostgreSQL та IoT-модуля. Усі компоненти взаємодіють між собою через чітко визначені інтерфейси та API.
Мобільний застосунок є основним інтерфейсом взаємодії користувача із системою. Користувач виконує авторизацію, переглядає інформацію про підписки, здійснює пошук бізнесів, обирає тарифні плани та переглядає історію платежів. Мобільний клієнт формує HTTP-запити та надсилає їх на backend-сервер через REST API.
Backend-сервер приймає запити від клієнта, виконує оброблення даних та звертається до сервісного шару. Сервіси відповідають за реалізацію бізнес-логіки: створення бізнесів, управління шаблонами підписок, оформлення підписок користувачів, обчислення статистики та перевірку ролей доступу. Після оброблення інформації сервер формує JSON-відповідь і надсилає її назад мобільному клієнту.
Для зберігання даних backend використовує базу даних PostgreSQL. Репозиторії Spring Data JPA забезпечують доступ до таблиць бази даних та виконують CRUD-операції. Наприклад, під час створення нової підписки сервер записує інформацію до таблиці user_subscription, а при створенні нового бізнесу — до таблиці business.
Окрему роль у системі відіграє IoT-компонент. IoT-пристрої можуть фіксувати використання певних сервісів або ресурсів користувачем. Наприклад, фітнес-зал може автоматично реєструвати відвідування клієнта через IoT-пристрій. Після цього інформація надсилається на backend, де створюється запис у таблиці iot_usage_log, а також оновлюються дані про використання компонентів підписки.
Між компонентами системи забезпечується постійна взаємодія. Мобільний застосунок не працює напряму з базою даних, а отримує всю інформацію через backend API. Це підвищує рівень безпеки, дозволяє централізовано контролювати бізнес-логіку та спрощує подальше масштабування системи.

3.3 Опис діаграми прецедентів
Діаграма прецедентів відображає основні сценарії взаємодії користувачів із системою FlexiSub. У системі присутні чотири основні актори: звичайний користувач, бізнес-користувач, адміністратор та IoT-пристрій.
Звичайний користувач може реєструватися, авторизовуватися, переглядати список бізнесів, обирати підписки, переглядати активний тарифний план та історію платежів. Бізнес-користувач має можливість створювати та редагувати шаблони підписок, переглядати клієнтів і аналітику. Адміністратор відповідає за перевірку та підтвердження бізнесів. IoT-пристрої автоматично передають дані про використання сервісів до системи.
 
Рис 3.3 - опис діаграми прецедентів

3.4 Опис діаграми компонентів
Діаграма компонентів демонструє архітектуру системи та взаємозв’язки між її основними модулями. Мобільний застосунок складається з набору Activity-класів, які відповідають за окремі екрани застосунку. Backend-система включає контролери, сервіси та репозиторії.
Контролери приймають HTTP-запити від клієнта та передають їх до сервісного шару. Сервіси реалізують бізнес-логіку системи, а репозиторії виконують доступ до бази даних. База даних PostgreSQL використовується як центральне сховище всієї інформації системи.
IoT-компонент взаємодіє з backend через окремий API та передає інформацію про використання компонентів підписки.
 
Рис 3.4 - опис діаграми компонентів


3.5 Опис діаграми пакетів
Діаграма пакетів використовується для відображення логічного розділення системи на окремі модулі. У мобільному застосунку виділено пакети для UI, API-клієнта, моделей даних та сервісів авторизації.
У backend-системі виділено пакети controllers, services, repositories, entities та dto. Таке розділення дозволяє підтримувати чисту архітектуру проєкту та спрощує супровід коду.
Пакет controllers відповідає за прийом HTTP-запитів, services містить бізнес-логіку, repositories реалізує доступ до бази даних, entities описує структуру таблиць, а dto використовується для передачі даних між клієнтом та сервером.
 
Рис 3.5 - діаграми пакетів
3.6 Опис діаграми діяльності
Діаграма діяльності описує процес роботи користувача із системою після запуску застосунку. Користувач вводить логін і пароль, після чого мобільний застосунок надсилає запит авторизації на backend-сервер.
У разі успішної авторизації система визначає роль користувача та відкриває відповідний екран. Якщо користувач має роль BUSINESS, відкривається бізнес-панель із доступом до статистики, шаблонів підписок і клієнтів. Якщо роль USER, відкривається користувацький dashboard із інформацією про активну підписку та доступні плани.
У випадку помилки авторизації система відображає повідомлення про помилку та пропонує повторити введення даних.
 
Рис 3.6 - діаграми пакетів
3.7 Архітектура Android-застосунку
Архітектура Android-застосунку визначає структуру програмної системи, принципи взаємодії її компонентів та організацію оброблення даних. Основною метою архітектури є забезпечення зручності підтримки, масштабованості, повторного використання коду та розділення відповідальності між окремими модулями системи.
Сучасні Android-застосунки зазвичай будуються за багаторівневою архітектурою, у якій окремо виділяються рівень інтерфейсу користувача, рівень бізнес-логіки та рівень доступу до даних. Такий підхід дозволяє ізолювати логіку оброблення даних від елементів інтерфейсу та спрощує тестування й подальшу модифікацію застосунку.
Рівень інтерфейсу користувача відповідає за взаємодію з користувачем, відображення екранів, оброблення натискань кнопок та введення даних. Для побудови інтерфейсу використовуються XML-layout файли та Activity або Fragment компоненти.
Рівень роботи з даними забезпечує взаємодію із сервером або локальним сховищем. Для надсилання HTTP-запитів часто використовуються спеціалізовані бібліотеки, які дозволяють виконувати обмін даними через REST API та працювати з JSON-форматом.
Бізнес-логіка застосунку відповідає за оброблення даних, перевірку умов, управління сценаріями роботи та взаємодію між окремими компонентами системи. Завдяки розділенню логіки та інтерфейсу забезпечується більш зрозуміла структура проєкту.
Для збереження інформації про користувача, налаштування або токени доступу можуть використовуватись локальні механізми зберігання даних, наприклад SharedPreferences або локальні бази даних.
Таким чином, архітектура Android-застосунку забезпечує модульність, спрощує розширення функціоналу та підвищує стабільність роботи програмної системи.
3.8 Використання засобів ШІ
Під час розробки мобільного застосунку FlexiSub використовувалися засоби штучного інтелекту для прискорення процесу створення програмного забезпечення, генерації коду, проєктування архітектури системи та підготовки документації. Використання ШІ дозволило швидше реалізувати окремі функціональні модулі, оптимізувати структуру Android-застосунку та спростити створення UML-діаграм і текстових описів.

•	генерації шаблонів Android Activity;
•	створення XML-layout інтерфейсів;
•	генерації REST API endpoint-ів;
•	створення DTO-моделей;
•	реалізації взаємодії з Retrofit API;
•	побудови UML-діаграм;
•	генерації SQL-структури бази даних;
•	написання технічної документації та описів архітектури;
•	формування прикладів коду для backend та frontend частини системи.
Під час роботи використовувалися текстові запити (prompts), які описували необхідну функціональність або структуру компонентів системи. На основі введених запитів система ШІ генерувала приклади коду, архітектурні рішення та рекомендації щодо реалізації функціоналу.
