В цій лабораторній роботі необхідно показати як можна масштабувати бекенд системи для роботи із великим навантаженням. Для цього, можна на вибір: 
-	масштабувати сервер горизонтально — багато копій сервера виконують однакові функції для різних користувачів; 
-	масштабувати сервер вертикально — різні мікросервіси виконують різні функції і масштабуються окремо одне від одного.  
З технічної точки зору, масштабування можна виконати так: 
-	деплой сервера на Kubernetes кластер, разом із окремим load balancer-ом та базою даних. При цьому треба буде показати коректну роботу системи при збільшенні чи зменшенні кількості подів сервера чи окремих мікросервісів; 
-	деплой в serverless середовище (AWS Lambda, Firebase Cloud Functions або аналогічне). Треба показати що система може зменшувати кількість активних інстансів до 0, збільшувати її зі збільшенням трафіку і все ще коректно функціонувати. 
-	ваш власний спосіб показати що система може масштабуватись. 
 	 
3 ОПИС ВИКОНАНОЇ РОБОТИ 
 
3.1 Стратегія масштабування системи 
Для системи Subscription Platform було обрано горизонтальне масштабування backend API. Основна ідея такого підходу полягає в запуску декількох однакових екземплярів серверної частини замість збільшення ресурсів одного сервера. Кожен pod виконує однакову бізнес-логіку та може незалежно обробляти HTTP-запити користувачів.
У системі Subscription Platform користувачі взаємодіють із сервісом через веб-інтерфейс, виконуючи авторизацію, управління компаніями, тарифними планами, підписками та платежами. При збільшенні кількості одночасних користувачів навантаження на серверну частину зростає, тому для забезпечення стабільної роботи використовується Kubernetes.
Kubernetes Service виконує роль балансувальника навантаження. Він надає єдину точку доступу до API, а всі вхідні HTTP-запити автоматично розподіляються між доступними pod-ами backend-застосунку. Завдяки цьому користувачі працюють з одним endpoint незалежно від фактичної кількості серверів.
База даних PostgreSQL у даній роботі залишається єдиною точкою зберігання даних. Масштабування бази даних не розглядається, оскільки потребує окремих механізмів реплікації та кластеризації.
Стратегія масштабування включає:
– горизонтальне масштабування Spring Boot API через параметр replicas у Kubernetes Deployment;
– використання Kubernetes Service для балансування HTTP-запитів;
– винесення PostgreSQL в окремий сервіс;
– можливість зміни кількості replicas через kubectl scale або Kubernetes Dashboard;
– використання Horizontal Pod Autoscaler для автоматичного масштабування;
– виконання навантажувального тестування для оцінки продуктивності системи.

3.5 Опис навантажувального тестування 
Для перевірки ефективності масштабування було виконано навантажувальне тестування системи Subscription Platform.
Тестування проводилося за наступним сценарієм:
– запуск Kubernetes-кластера;
– розгортання системи через файл kubernetes.yaml;
– встановлення кількості replicas = 1;
– виконання серії HTTP-запитів до REST API;
– повторення тестування для replicas = 3 та replicas = 5;
– аналіз показників продуктивності.
Під час тестування вимірювалися:
– кількість запитів за секунду (RPS);
– середній час відповіді;
– кількість помилок;
– стабільність роботи системи.

4 ВИСНОВКИ 
У ході виконання лабораторної роботи було досліджено можливості масштабування веб-застосунку Subscription Platform за допомогою Kubernetes. Було реалізовано контейнеризацію серверної частини на базі Docker, налаштовано Kubernetes Deployment, Service та Horizontal Pod Autoscaler.
В результаті виконаної роботи було забезпечено можливість горизонтального масштабування Spring Boot API шляхом запуску декількох копій застосунку. Kubernetes Service забезпечує балансування навантаження між pod-ами, а Horizontal Pod Autoscaler автоматично змінює кількість екземплярів сервера залежно від навантаження на процесор.
Проведене навантажувальне тестування підтвердило, що зі збільшенням кількості replicas система здатна обробляти більшу кількість запитів за секунду при зменшенні часу відповіді. Отримані результати підтверджують ефективність використання Kubernetes для побудови масштабованих веб-застосунків та демонструють готовність Subscription Platform до роботи зі значною кількістю одночасних користувачів.
